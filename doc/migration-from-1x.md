# Migrating from 1.x to 2.0.0

Version 2.0.0 is a full rewrite. It keeps the same algorithm and the same code
output — **codes generated by 1.x and 2.0 from the same secret are identical** —
but it tightens the API: fields are now immutable, inputs are validated with
real exceptions, and the `otpauth://` URL format was corrected to match Google
Authenticator. This page lists every breaking change with before/after code, and
then the new features you get once you have migrated.

The single most important reassurance up front: **your users do not need to
re-enroll.** The secret is unchanged, so every already-provisioned token keeps
working. The only things that change are *how your Dart code is written* and what
a *newly generated* QR looks like.

For the canonical change list see the `[2.0.0]` section of
[CHANGELOG.md](../CHANGELOG.md). For the new API in depth, see
[Getting started](./getting-started.md), the [TOTP guide](./totp.md) and the
[HOTP guide](./hotp.md).

## At a glance

| # | Change | Likely impact |
| - | --- | --- |
| 1 | Dart SDK floor raised to `>=3.0.0 <4.0.0` | Update your SDK constraint. |
| 2 | `digits`, `secret`, `algorithm` are now `final` | Stop mutating fields after construction. |
| 3 | `TOTP.interval` is non-nullable `final int` | Drop `!` on `interval`; it is always set. |
| 4 | Invalid inputs throw `ArgumentError` (release too) | Wrap construction in try/catch, or validate first. |
| 5 | Base32 secret validated at construction | Bad secrets fail *earlier*, on `new`, not on generate. |
| 6 | `otpauth://` label is now `Issuer:Account` | Newly generated QR codes look different (existing tokens unaffected). |
| 7 | `Util.intToBytelist` `padding` -> `byteLength` | Rename the argument; semantics corrected to big-endian. |
| 8 | Util methods take `required`, return non-nullable | Drop `!` and named-arg juggling on internal helpers. |

Each is detailed below.

## 1. Dart SDK 3.0 floor

2.0.0 requires Dart 3. Update your `pubspec.yaml`:

```yaml
# Before
environment:
  sdk: '>=2.12.0 <3.0.0'

# After
environment:
  sdk: '>=3.0.0 <4.0.0'
```

If you cannot move to Dart 3 yet, stay on the 1.3.x line.

## 2. Fields are now `final`

In 1.x, `digits`, `secret` and `algorithm` were mutable, so you could reassign
them after construction. In 2.0 they are `final`: set them in the constructor
and never touch them again. To "change" a token, build a new one.

```dart
// Before (1.x) — mutation allowed
final totp = TOTP(secret: 'J22U6B3WIWRRBTAV');
totp.digits = 8;            // compiled, mutated the live token
totp.secret = 'NEWSECRET';  // compiled

// After (2.0) — fields are final; reassignment is a compile error
final totp = TOTP(secret: 'J22U6B3WIWRRBTAV', digits: 8);
// totp.digits = 8;  // ERROR: 'digits' can't be used as a setter

// Build a fresh token instead of mutating:
final wider = TOTP(secret: totp.secret, digits: 8, interval: totp.interval);
```

This is almost always a latent-bug fix: mutating a token's secret or digit count
mid-flight was rarely intentional.

## 3. `TOTP.interval` is non-nullable

In 1.x `interval` was declared `int?` and was sprinkled with `!` internally. In
2.0 it is a plain `final int`, defaulting to 30, and the constructor rejects any
value that is not positive.

```dart
// Before (1.x)
final TOTP totp = TOTP(secret: 'J22U6B3WIWRRBTAV');
final int period = totp.interval!;   // had to null-assert

// After (2.0)
final totp = TOTP(secret: 'J22U6B3WIWRRBTAV');
final int period = totp.interval;    // always set, no `!`

// A non-positive interval now throws at construction:
// TOTP(secret: 'J22U6B3WIWRRBTAV', interval: 0); // ArgumentError
```

Remove any `!` you applied to `interval` and any null checks around it.

## 4. Invalid arguments throw `ArgumentError`

1.x guarded inputs with `assert`, which is **stripped in release builds** — so a
`digits: 10` token that crashed in debug would silently produce wrong-length
codes in production. 2.0 validates in the constructor and throws `ArgumentError`
in **every** build mode.

What is validated, and the rule:

| Field | Rule |
| --- | --- |
| `digits` | between 6 and 8 |
| `secret` | non-empty, decodable Base32, decodes to >= 1 byte |
| `interval` (TOTP) | positive |
| `counter` (HOTP) | non-negative |
| `window` (`verify`) | non-negative |

```dart
// Before (1.x) — assert only; passed silently in release
final totp = TOTP(secret: 'J22U6B3WIWRRBTAV', digits: 10); // wrong, but ran

// After (2.0) — throws in all build modes
try {
  final totp = TOTP(secret: 'J22U6B3WIWRRBTAV', digits: 10);
} on ArgumentError catch (e) {
  // handle invalid configuration
}
```

If you construct tokens from user-supplied or stored values, wrap construction in
a `try`/`catch` (or validate the inputs first).

## 5. Base32 secret validation moved to construction

This is the subtlest change, so it gets its own section.

In 1.x the secret was only Base32-decoded *the first time you generated a code*.
Two consequences followed, both fixed in 2.0:

- **A malformed secret crashed late.** `TOTP(secret: 'not valid base32!')`
  constructed fine and only blew up later inside `now()`/`verify()`, often far
  from the code that supplied the bad secret. In 2.0 the constructor decodes the
  secret immediately and throws `ArgumentError` right there.

- **Too-short secrets are now rejected.** Strings shorter than one Base32 group,
  such as `"ABC"`, decode to **zero bytes** — an empty, insecure HMAC key. 1.x
  accepted them and produced codes from an empty key; 2.0 rejects them with an
  `ArgumentError` ("too short to decode to a Base32 key").

```dart
// Before (1.x)
final totp = TOTP(secret: 'lowercase-or-junk'); // constructs fine
final code = totp.now();                          // crashes here, later

// After (2.0)
final totp = TOTP(secret: 'lowercase-or-junk');   // throws ArgumentError now
```

**About lowercase secrets:** the secret must be the **uppercase** RFC 4648
Base32 alphabet (`A`–`Z`, `2`–`7`). If you had a lowercase secret that "seemed to
work" in 1.x, it did not really — it either decoded to different bytes than the
authenticator app used (so the codes never actually matched a real app) or
happened to decode by luck on a permissive decoder. There is no behaviour to
preserve here. **Normalise secrets to uppercase** before constructing:

```dart
final totp = TOTP(secret: rawSecret.toUpperCase().trim());
```

If you generate secrets with the new `OTP.randomSecret()`, they are already in
the correct uppercase alphabet.

## 6. `otpauth://` URL label is now `Issuer:Account`

1.x put only the account in the label and double-encoded the issuer oddly. 2.0
follows the Google Authenticator
[Key URI Format](https://github.com/google/google-authenticator/wiki/Key-Uri-Format):
when you pass an `issuer`, the label becomes `Issuer:Account`, and **all** query
parameters are URL-encoded.

```dart
final totp = TOTP(secret: 'J22U6B3WIWRRBTAV');
final url = totp.generateUrl(issuer: 'Acme', account: 'alice@example.com');

// Before (1.x) — account only in the label
// otpauth://totp/alice%40example.com?secret=J22U6B3WIWRRBTAV&issuer=Acme&...

// After (2.0) — Issuer:Account label, all params encoded
// otpauth://totp/Acme:alice%40example.com?secret=J22U6B3WIWRRBTAV&issuer=Acme&digits=6&algorithm=SHA1&period=30
```

**Impact:** a QR code you *regenerate* with 2.0 will look different (the label
changed) and will display the issuer/account more nicely in authenticator apps.
**Already-enrolled tokens are completely unaffected** — the secret in the URL is
identical, and that is the only thing that determines the codes. You do not need
to re-provision anyone; the label is cosmetic metadata.

See [otpauth:// URIs](./otpauth-uri.md) for the full format.

## 7. `Util.intToBytelist`: `padding` -> `byteLength`

> This only affects you if you called the **internal** `Util` helpers directly
> (`package:dart_dash_otp/src/utils/...`). It is not part of the exported API.

The parameter was renamed from `padding` to `byteLength`, and the implementation
was rewritten to emit a correct big-endian byte sequence of exactly that length.
(The 1.x version shifted by the `padding` value rather than by 8 bits per byte,
which was only correct by accident at the default of 8.)

```dart
import 'package:dart_dash_otp/src/utils/generic_util.dart';

// Before (1.x)
final bytes = Util.intToBytelist(1, padding: 8);

// After (2.0) — renamed; now a true big-endian length
final bytes = Util.intToBytelist(1, byteLength: 8);
// => [0, 0, 0, 0, 0, 0, 0, 1]
```

`byteLength` must be positive or it throws `ArgumentError`.

## 8. Util methods: required, non-nullable

> Also internal-only.

`AlgorithmUtil.createHmacFor`, `AlgorithmUtil.rawValue` and
`OTPUtil.otpTypeValue` now take `required`, non-nullable parameters and return
non-nullable values. In 1.x `createHmacFor` returned a nullable `Hmac?` that
callers had to `!`-assert.

```dart
import 'package:crypto/crypto.dart';
import 'package:dart_dash_otp/src/components/otp_algorithm.dart';
import 'package:dart_dash_otp/src/utils/algorithm_util.dart';

// Before (1.x) — nullable return, needed `!`
final Hmac hmac = AlgorithmUtil.createHmacFor(
  algorithm: OTPAlgorithm.SHA1,
  key: keyBytes,
)!;

// After (2.0) — non-nullable, no `!`
final Hmac hmac = AlgorithmUtil.createHmacFor(
  algorithm: OTPAlgorithm.SHA1,
  key: keyBytes,
);
```

Drop the `!` and remove any null handling around these helpers.

## What did NOT change

To keep the diff in your head small:

- **Code output is identical.** Same secret in, same codes out. No re-enrollment.
- **`OTPAlgorithm` cases stay `SHA1`/`SHA256`/`SHA384`/`SHA512`** (deliberately
  kept in SCREAMING_CASE for 1.x source compatibility).
- **`OTPType` is still `TOTP`/`HOTP`.**
- The TOTP/HOTP constructors keep the same parameter names and defaults
  (`digits: 6`, `interval: 30`, `counter: 0`, `algorithm: SHA1`).

## New in 2.0 — available once you have migrated

The rewrite is not only restrictions; it adds the features the 1.x API was
missing. None of these require changes to *existing* calls — they are new
capabilities to adopt when you want them.

- **Drift-tolerant verification.** `TOTP.verify(otp: ..., window: 1)` and
  `HOTP.verify(otp: ..., counter: ..., window: 3)` accept neighbouring time
  steps / look-ahead counters. (Mind the trade-offs in
  [Security considerations](./security-considerations.md#window-trade-offs).)
- **Constant-time comparison.** Both `verify` paths compare codes in constant
  time, closing the timing side-channel that 1.x's `otp == expected` opened.
- **Provisioning from a scanned QR.** `TOTP.fromUri(uri)` and `HOTP.fromUri(uri)`
  parse an `otpauth://` URL into a token. See [otpauth:// URIs](./otpauth-uri.md).
- **Secure secret generation.** `OTP.randomSecret()` returns a 160-bit
  (RFC 4226-recommended) Base32 secret from `Random.secure`.
- **Countdown support.** `TOTP.remainingSeconds()` returns how long the current
  code stays valid — ideal for a countdown ring. See the
  [Flutter integration](./flutter-integration.md) guide.

## Migration checklist

- [ ] Bump SDK constraint to `>=3.0.0 <4.0.0`.
- [ ] Remove field reassignments (`digits`/`secret`/`algorithm` are `final`).
- [ ] Drop `!` on `TOTP.interval`.
- [ ] Wrap token construction in `try`/`catch` for `ArgumentError`, or validate
      inputs first.
- [ ] Normalise stored secrets to **uppercase** before constructing.
- [ ] Re-check any code that depended on the old `otpauth://` label shape.
- [ ] If you used internal `Util`/`AlgorithmUtil`/`OTPUtil` helpers, rename
      `padding` -> `byteLength` and drop now-unnecessary `!`.
- [ ] Consider adopting `window`, `fromUri`, `randomSecret` and
      `remainingSeconds`.
